# 面试问答准备

## 1. 这个项目解决什么问题？

这个项目解决的是自动化测试失败后的分析效率问题。传统自动化测试只能告诉我们“哪个用例失败了”，但排查时还要人工去看日志、断言、接口响应和截图。这个项目把失败上下文收集成结构化 JSON，再用 AI 生成缺陷诊断报告，帮助快速判断是业务 bug、测试脚本问题还是环境问题。

## 2. 为什么要自己写一个 FastAPI Demo 系统？

因为只写测试脚本不容易展示完整测试工程能力。自己构建被测系统后，可以稳定复现登录、商品查询、下单、库存不足等场景，也能控制测试数据和失败样例。这样面试官能看到从被测服务、自动化测试、CI 到失败分析的完整闭环。

## 3. pytest 在项目里负责什么？

pytest 是统一测试入口。项目里用它跑三类测试：

- 服务层单元测试，验证登录、库存、下单等业务逻辑。
- API 测试，验证 HTTP 状态码、响应体和鉴权行为。
- Playwright E2E 测试，验证真实浏览器里的登录和下单流程。

同时 `tests/conftest.py` 里使用 pytest hook，在用例失败时自动保存失败 JSON 产物。

## 4. Playwright 在项目里负责什么？

Playwright 负责浏览器端到端测试。它不是只调接口，而是真的启动浏览器访问页面，完成登录、进入商品页、创建订单等用户流程。这样可以发现接口正常但页面流程异常的问题。

## 5. AI 模块怎么工作？

AI 模块分成四步：

1. 读取 `reports/latest/failures/*.json` 里的失败产物。
2. 把失败用例、报错、耗时和标签组装成 prompt。
3. 如果配置了 `OPENAI_API_KEY`，调用 OpenAI 生成诊断报告。
4. 如果没有 API key 或调用失败，生成 fallback 报告，不影响测试和 CI。

## 6. 为什么 AI 调用失败不能让 CI 失败？

因为自动化测试本身才是核心质量门禁，AI 是辅助分析能力。如果 AI 服务超时、限流、没有配置 key，CI 仍然应该保留 pytest 报告和失败产物，不能因为辅助功能失败而遮住真正的测试结果。

## 7. 这个项目体现了哪些自动化测试能力？

- pytest fixture 管理测试数据库
- 参数清晰的 API 测试
- Playwright 浏览器 E2E 测试
- 测试失败产物收集
- pytest-html 报告
- GitHub Actions CI
- 失败场景样例和可复现 demo flow

## 8. 这个项目体现了哪些 AI 应用开发能力？

- 把 AI 放入真实业务流程，而不是单独做聊天 demo
- 设计 prompt，把测试失败上下文转成结构化输入
- 处理 API key 缺失、网络异常、AI 调用失败等工程问题
- 输出 Markdown 报告，方便 CI 上传和人工阅读
- 保持 AI 模块和测试框架解耦

## 9. 如果继续扩展，你会做什么？

可以继续扩展三个方向：

- 增加历史趋势 dashboard，展示通过率和失败原因分类。
- 接入真实截图/trace 摘要，让 AI 报告包含更多证据。
- 对接 GitHub PR comment，在 CI 失败时自动评论诊断结果。

## 10. 面试时一句话总结

这是一个面向求职展示的 AI 自动化测试平台：它用 Python 完成真实 Web/API 测试工程流程，并把 AI 接入失败分析环节，展示我从自动化测试向 AI 应用开发迁移的能力。
